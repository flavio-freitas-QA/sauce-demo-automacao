# Dia 004 — Testes de API (Postman/Newman)

## Contexto

Este dia expande a cobertura de testes de API do projeto, utilizando duas APIs públicas de teste para cobrir cenários que não seriam possíveis com o Sauce Demo, que não expõe uma API própria estável para automação.

A proposta é validar:
- CRUD de recursos em JSONPlaceholder
- Relacionamentos entre recursos
- Autenticação simulada e criação de usuários em ReqRes
- Paginação e cenários negativos

Importante: o bloco de autenticação do ReqRes foi implementado de forma compatível com o comportamento atual da API pública, que pode negar as requisições por configuração ou credenciais. O objetivo é validar o contrato da API e o comportamento esperado de erro, sem assumir que o login bem-sucedido será sempre possível em ambientes externos.

## Objetivo

Implementar testes de API via Postman/Newman cobrindo 16 cenários distribuídos entre duas collections:
- JSONPlaceholder: CRUD, relacionamentos e validação de schema
- ReqRes: autenticação simulada, registro e paginação

## Escopo previsto

### Collection 1 — JSONPlaceholder
Arquivo esperado:
- postman/collections/dia-004-jsonplaceholder.postman_collection.json

Environment esperado:
- postman/environments/jsonplaceholder.postman_environment.json

Cenários:
1. POST /posts — criar recurso
2. GET /posts/{id} — buscar recurso criado
3. PUT /posts/{id} — atualizar recurso
4. DELETE /posts/{id} — deletar recurso
5. GET /posts/99999 — recurso inexistente (404)
6. POST /posts — corpo malformado/campos faltando
7. GET /posts/{id}/comments — relacionamento post → comentários
8. GET /users — listar usuários com validação de schema
9. GET /users/1 — buscar usuário específico com estrutura aninhada
10. GET /users/1/todos — relacionamento usuário → tarefas

### Collection 2 — ReqRes
Arquivo esperado:
- postman/collections/dia-004-reqres-auth.postman_collection.json

Environment esperado:
- postman/environments/reqres.postman_environment.json

Cenários:
11. POST /api/login — credenciais válidas
12. POST /api/login — credenciais inválidas
13. POST /api/register — dados válidos
14. GET /api/users?page=2 — validação de paginação
15. GET /api/users/2 — buscar usuário específico
16. GET /api/users/9999 — usuário inexistente (404)

Observação: quando a API pública negar autenticação por configuração/segurança, os testes validam corretamente o retorno de erro e a estrutura da resposta, sem afirmar falsamente que o fluxo de login foi executado com sucesso.

## Regras de implementação

### Estrutura das collections
- Separar os cenários em pastas lógicas para facilitar leitura e manutenção.
- Reaproveitar o environment de JSONPlaceholder já existente.
- Criar um novo environment para ReqRes com a base URL configurada.

### Assertions obrigatórias
Cada request deve validar:
- Status code esperado
- Tipo de dado retornado (string, number, array, object, boolean)
- Campos esperados na resposta
- Captura de variáveis quando necessário, como:
  - postId
  - token

### Atenção especial ao ReqRes
A API ReqRes pode exigir o header x-api-key em requisições recentes. Antes de implementar os testes, validar a documentação oficial atual para confirmar se o header é necessário, qual valor usar e se há alguma mudança de comportamento.

## Scripts npm e CI

Os scripts esperados são:

```json
"api:test:jsonplaceholder": "newman run postman/collections/dia-004-jsonplaceholder.postman_collection.json -e postman/environments/jsonplaceholder.postman_environment.json",
"api:test:reqres": "newman run postman/collections/dia-004-reqres-auth.postman_collection.json -e postman/environments/reqres.postman_environment.json",
"api:test": "npm run api:test:jsonplaceholder && npm run api:test:reqres"
```

O workflow de CI deve executar o comando:
- npm run api:test

## Validação esperada

- Executar localmente o comando npm run api:test
- Confirmar que as duas collections passam sem falhas
- Garantir que não haja segredos sensíveis hardcoded em arquivos versionados

## Problemas e descobertas esperadas

- A API JSONPlaceholder é um mock e pode comportar-se de forma diferente de uma API real; por exemplo, um POST malformado pode retornar 201 em vez de 400.
- O ReqRes pode exigir headers adicionais recentes, como x-api-key, o que pode causar falhas iniciais se a implementação não considerar essa exigência.
- A separação em duas APIs públicas é intencional, pois cada uma cobre um conjunto distinto de cenários de teste.

## Decisão técnica

A abordagem de usar duas APIs diferentes é justificada porque:
- o Sauce Demo não expõe uma API própria para automação
- JSONPlaceholder cobre CRUD e relacionamentos
- ReqRes cobre autenticação simulada, registro e paginação

Essa divisão permite uma cobertura mais ampla e realista do que usar uma única API pública para todos os cenários.
